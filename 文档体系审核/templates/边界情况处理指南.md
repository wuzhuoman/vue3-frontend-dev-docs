## 代码变更处理

### 场景：审核期间代码被更新

**检测方法**：
```bash
# 每日审核前执行
git log -1 --format="%H %ai"
# 对比与初始commit hash是否一致
```

**处理流程**：

1. **记录变更信息**
   - 在`审核日志.md`中记录新的commit hash和变更内容
   - 执行`git log -5`查看最近的变更记录

2. **评估影响范围**
   - 分析变更涉及的文件和模块
   - 判断是否影响已审核的文档：
     - ✅ API接口变更 → 重新审核相关API文档
     - ✅ 组件结构变更 → 重新审核相关组件文档
     - ✅ 工具函数变更 → 重新审核工具函数文档
     - ✅ 配置文件变更 → 重新审核配置相关文档

3. **重新审核受影响文档**
   - 将受影响的文档状态从"已完成"改为"需重新审核"
   - 在审核状态记录表中记录"需重新审核"状态
   - 按优先级重新审核这些文档

4. **更新审核进度**
   - 调整阶段进度百分比
   - 在审核报告中记录代码更新情况

**预防措施**：
- ⚠️ 审核期间避免执行`git pull`
- ⚠️ 保持代码版本稳定
- ⚠️ 如必须更新，先完成当前阶段审核

## 文档变更处理

### 场景：被审核的文档在审核期间被修改

**检测方法**：
- 每次审核文档前，检查文档的最后修改时间
- 对比文档修改时间与审核开始时间

**处理流程**：

1. **暂停当前审核**
   - 立即停止对该文档的审核
   - 保存已审核的部分（如有）

2. **对比修改内容**
   - 使用Git查看文档变更：`git diff dev_docs/xxx.md`
   - 分析修改是否影响审核结果：
     - ✅ 格式调整 → 不影响，继续审核
     - ✅ 内容修改 → 重新审核
     - ✅ 新增内容 → 检查是否需要审核新增部分

3. **记录变更情况**
   - 在`审核日志.md`中记录文档修改情况
   - 说明修改的文件、时间和内容范围

4. **重新审核**
   - 从文档开头重新审核
   - 特别关注修改的部分
   - 确保审核结果的准确性

5. **更新审核状态**
   - 将文档审核时间更新为重新审核的时间
   - 在审核状态记录表中备注"重新审核"

## Git异常处理

### 场景1：Git分支状态为Dirty（有未提交更改）

**检测方法**：
```bash
git status --porcelain
# 如果有输出，说明有未提交的更改
```

**处理方式**：

**选项A：先提交再审核（推荐）**
```bash
# 提交所有更改
git add .
git commit -m "审核前提交代码"

# 然后开始审核
```

**选项B：继续审核并记录**
- 在`审核报告.md`中注明"Git状态：Dirty"
- 在"代码快照信息"表格中列出所有未提交的文件：
  ```markdown
  | 未提交文件 | 状态 |
  |-----------|------|
  | src/api/user.ts | M (已修改) |
  | src/components/New.vue | A (已添加) |
  | src/utils/old.js | D (已删除) |
  ```
- 在审核日志中说明未提交的原因
- ⚠️ **注意**：Dirty状态可能导致审核结果不准确

---

### 场景2：审核期间切换Git分支

**处理流程**：

1. **完成当前文档审核**
   - 先完成当前正在审核的文档
   - 确保审核结果完整

2. **切换分支**
   - 执行`git checkout <branch-name>`
   - 切换后重新获取代码快照信息

3. **评估影响**
   - 分析分支切换对已审核文档的影响
   - 不同分支可能有不同的代码实现
   - 可能需要重新审核部分文档

4. **更新审核信息**
   - 在`审核报告.md`中更新"代码快照信息"
   - 记录分支切换的时间和原因
   - 在审核日志中说明分支切换情况

**建议**：
- ⚠️ 审核期间避免切换Git分支
- ⚠️ 如必须切换，建议分两次审核

## AI能力边界处理

### 场景1：AI无法确定修复方案

**识别标准**：
- 问题涉及复杂的业务逻辑
- 有多种可能的修复方案
- AI不确定哪种方案最优
- 修复可能影响其他模块

**处理流程**：

1. **记录问题并标注**
   - 在`问题清单.md`中创建"需人工审核"部分
   - 详细说明问题的复杂性和不确定性
   - 列出AI考虑的多种可能的修复方案

2. **创建临时建议文件**
   ```markdown
   # 临时修复建议：[问题描述]

   ## 问题背景
   ...

   ## AI考虑的修复方案

   ### 方案1：[方案名称]
   - 优点：...
   - 缺点：...
   - 可能影响：...

   ### 方案2：[方案名称]
   - 优点：...
   - 缺点：...
   - 可能影响：...

   ## AI建议
   建议选择方案X，原因：...

   ## 需要开发者决策
   请开发者根据实际情况选择合适的修复方案。
   ```

3. **不强行修改**
   - ❌ 不使用`replace_in_file`修改文档
   - ❌ 不标注为"已修复"
   - ✅ 标注为"需人工审核"
   - ✅ 等待开发者反馈

4. **后续处理**
   - 待开发者决策后，由AI执行修复
   - 或由开发者手动修复，AI仅负责记录

---

### 场景2：AI工具调用受限

**识别标准**：
- 工具调用次数达到限制
- Token使用超限
- 网络连接问题导致工具调用失败

**处理流程**：

1. **优先完成高优先级文档**
   - 暂停低优先级文档的审核
   - 优先审核优先级1和优先级2的文档

2. **保存当前进度**
   - 在`审核日志.md`中详细记录当前审核进度：
     ```markdown
     ### 工具受限情况
     - 受限时间：YYYY-MM-DD HH:mm:ss
     - 当前审核文档：xxx.md
     - 审核进度：约50%
     - 已发现但未处理的问题：X个
     - 下一步计划：待工具限制解除后继续审核
     ```

3. **更新审核状态**
   - 在`审核状态记录表.md`中将当前文档状态改为"进行中"
   - 添加备注："工具受限，暂停审核"

4. **等待恢复**
   - 工具限制解除后，阅读最新的审核日志
   - 从中断点继续审核
   - 确保审核结果的连续性

---

### 场景3：AI上下文容量不足

**识别标准**：
- 单个文档过长，超过AI上下文限制
- 需要大量代码对比时上下文不足
- 多文件并行审核时上下文溢出

**处理流程**：

1. **分批次审核大型文档**
   - 将长文档分为多个部分
   - 每次审核一个部分
   - 确保各批次之间有重叠部分，避免遗漏

2. **记录分批审核情况**
   - 在`审核日志.md`中说明分批审核：
     ```markdown
     ### 分批审核说明
     文档：xxx.md（过长，分3批审核）
     - 第1批：1-100行，时间：YYYY-MM-DD
     - 第2批：90-200行，时间：YYYY-MM-DD
     - 第3批：190-300行，时间：YYYY-MM-DD
     ```

3. **关联各批次审核结果**
   - 创建批次审核摘要
   - 确保问题不会在不同批次中重复
   - 最后生成整体的审核结果

## 审核范围调整

### 场景：只审核部分文档

**处理方式**：

1. **标记未审核文档**
   - 在`审核状态记录表.md`中，将不审核的文档状态标记为"已跳过"
   - 在"问题描述"列填写"不在本次审核范围内"

2. **更新审核报告**
   - 在`审核报告.md`的"审核范围"字段注明"部分文档审核"
   - 在"审核概况"表格中增加一列：
     ```markdown
     | 指标 | 数值 |
     |------|------|
     | 审核文档总数 | XX个（共YY个文档，ZZ个未审核） |
     | 完全准确 | XX个 (XX%) |
     | 已跳过 | ZZ个 |
     ```

3. **明确列出已审核和未审核文档**
   - 在审核报告中添加"审核范围清单"：
     ```markdown
     ## 📋 审核范围清单

     ### 已审核文档（XX个）
     - [x] architecture_overview.md
     - [x] multibrand_config.md
     - ...

     ### 未审核文档（XX个）
     - [ ] api_layer.md
     - [ ] stores_guide.md
     - ...
     ```

4. **阶段检查点调整**
   - 阶段检查点只记录已审核的文档
   - 说明哪些文档未审核及原因

## 版本不一致处理

### 场景：文档版本号与package.json不一致

**检测方法**：
1. 读取`package.json`获取当前版本号：
   ```bash
   cat package.json | grep "version"
   # 输出：  "version": "4.19.0",
   ```

2. 搜索文档中所有版本号引用：
   ```bash
   grep -r "v4\." dev_docs/
   ```

3. 对比发现不一致的版本号

**处理原则**：

**情况A：版本差异小于1个小版本（如v4.19.0 vs v4.19.1）**
- 直接更新文档中的版本号
- 标记为普通问题
- 在问题清单中记录

**情况B：版本差异较大（如v4.19.0 vs v5.0.0）**
- 记录为重要问题
- 提醒开发者确认文档是否需要重大更新
- 在问题清单中详细说明：
  ```markdown
  ### 版本号不一致问题

  **文档**：xxx.md
  **文档版本**：v4.19.0
  **实际版本**：v5.0.0

  **问题描述**：
  文档中引用的版本号与实际项目版本差异较大。

  **处理建议**：
  请开发者确认该文档是否需要更新到v5.0.0，或者版本号引用是否正确。
  ```

**操作步骤**：

1. **记录所有不一致**
   - 在问题清单中创建"版本不一致问题"部分
   - 列出所有发现的不一致项

2. **逐个处理**
   - 按处理原则逐个修复或上报
   - 对于简单差异，直接修复
   - 对于重大差异，记录并等待开发者确认

3. **验证修复**
   - 修复后再次搜索版本号
   - 确保所有引用都已更新

## 问题修复迭代

### 场景：修复一个问题后，引起新的问题

**处理流程**：

1. **记录新问题的发现**
   - 在`修复记录.md`中创建迭代记录：
     ```markdown
     ### 修复迭代记录

     **第1次修复**
     - 修复时间：YYYY-MM-DD HH:mm:ss
     - 修复问题：XXX
     - 修复方案：XXX

     **第2次修复**
     - 修复时间：YYYY-MM-DD HH:mm:ss
     - 修复问题：XXX（第1次修复引起）
     - 修复方案：XXX
     ```

2. **评估新问题的严重程度**
   - 如果新问题比原问题更严重，考虑回滚修复
   - 如果新问题严重程度相当，继续修复
   - 如果新问题较轻，可以在后续修复

3. **继续修复直到问题解决**
   - 重复修复过程
   - 确保最终修复不会引起新问题
   - 可能需要多次迭代

4. **最终验证**
   - 修复完成后，全面验证相关代码和文档
   - 确保没有遗漏的问题
   - 在问题清单中标记为"已修复"

**记录示例**：

```markdown
## 修复迭代案例

**原问题**：API接口文档中的示例代码参数错误

**第1次修复**：
- 修复时间：2026-01-15 10:30:00
- 修复方案：更新参数名称为正确的参数
- 修复结果：❌ 失败
- 新问题：参数类型错误，应该为字符串类型但示例中使用的是数字

**第2次修复**：
- 修复时间：2026-01-15 10:35:00
- 修复方案：修正参数类型为字符串
- 修复结果：❌ 失败
- 新问题：缺少必需的参数

**第3次修复**：
- 修复时间：2026-01-15 10:40:00
- 修复方案：添加缺失的必需参数
- 修复结果：✅ 成功
- 最终验证：通过
```

## 📞 遇到其他边界情况

如果遇到本指南未涵盖的边界情况：

1. **记录情况**
   - 在`审核日志.md`中详细描述遇到的情况
   - 说明问题的背景和当前状态

2. **保守处理**
   - 不强行修改文档
   - 保存当前进度
   - 标注为"需人工决策"

3. **寻求反馈**
   - 在问题清单中创建"特殊情况"部分
   - 详细说明情况并请求开发者指导

4. **更新指南**
   - 解决问题后，更新本指南
   - 添加新的边界情况处理方法
   - 持续完善边界情况处理体系
