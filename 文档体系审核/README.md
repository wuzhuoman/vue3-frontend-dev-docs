## 🚀 开始新的审核

### 步骤1：创建审核目录

在`文档体系审核`目录下创建新的审核子目录：

```
命名格式：YYYY-MM-DD-审核简述
示例：2026-01-12-文档全面审核
```

### 步骤2：复制模板文件

将`templates/`目录下的所有模板文件复制到新创建的审核目录中：

```bash
mkdir 2026-01-12-文档全面审核/checkpoints
copy templates\* 2026-01-12-文档全面审核\
```

### 步骤3：初始化审核状态

1. **获取代码快照信息**（重要！）
   参考`templates/代码快照获取指南.md`，执行Git命令获取当前代码状态，并填入`审核报告.md`的"代码快照信息"表格：

   ```bash
   # 获取当前Git分支
   git branch --show-current

   # 获取最新提交信息（commit hash + 提交消息 + 提交时间）
   git log -1 --format="%H %s %ai"

   # 检查是否有未提交的更改
   git status --porcelain
   # 如果返回空，说明分支状态为Clean；否则为Dirty

   # 获取项目版本号
   # 从package.json中读取"version"字段
   ```

   **为什么要记录Git信息？**
   - 多开发者联合开发项目，代码会持续演进
   - Git快照信息可以明确标注审核时对应的代码版本
   - 便于后续追溯问题，确认文档与代码的对应关系
   - 当代码更新后，可以根据Git信息判断是否需要重新审核文档

2. **备份初始状态**
   - 记录审核开始时的Git commit hash
   - 便于后续判断审核期间代码是否被修改

3. 编辑`审核报告.md`，填入上述获取的代码快照信息
4. 编辑`审核状态记录表.md`，更新日期和版本信息
   - **注意**：在审核状态记录表中，将`AI_Coding_Context.md`标记为"准备阶段阅读"（不参与正式审核）
5. 阅读`AI_Coding_Context.md`了解项目全局（**注意**：这是准备阶段，不作为正式审核文档）
6. 按照`文档审核方案.md`开始分阶段审核

## ✅ 审核完成标准

当满足以下条件时，审核可以认为完成：

1. **所有文档审核完毕**：`审核状态记录表.md`中所有文档状态为"已完成"
2. **问题修复完毕**：`修复记录.md`中所有严重和重要问题已修复
3. **生成审核报告**：`审核报告.md`包含完整的审核概况、问题清单和质量评估
4. **阶段检查点完整**：每个阶段的`checkpoints/`目录都有对应的阶段检查点文件

## 📊 问题处理原则

- **严重问题**：立即修复，24小时内完成
- **重要问题**：7天内修复
- **普通问题**：记录到修复计划，下次版本发布前修复

## 📝 审核注意事项

1. **先读AI_Coding_Context.md**：审核开始前必须先阅读此文档，了解项目全局
2. **基于代码审核**：所有审核结论必须基于实际代码，避免臆测
3. **及时更新记录**：审核过程中及时更新`审核状态记录表.md`和`审核日志.md`
4. **问题分类明确**：严格按照严重/重要/普通三级分类，便于后续修复
5. **保存检查点**：每个阶段完成后及时创建阶段检查点，便于断点续审

## 📅 定期审核建议

- **全面审核**：每季度进行一次
- **专项审核**：重大版本更新或架构调整后进行
- **快速复审**：每月进行一次快速复审，关注核心文档

## 🆘 常见问题

**Q1：审核中断后如何恢复？**
A1：
1. 查看最新的`审核状态记录表.md`，定位中断的文档
2. 阅读最近的阶段检查点，了解审核进度
3. 检查当前代码状态（`git log -1`）与审核开始时是否一致
4. 如果代码被更新，评估是否影响已审核的文档
5. 从中断位置继续审核

**Q2：发现代码和文档不一致时以哪个为准？**
A2：以实际代码为准，更新文档内容。如果认为代码有问题，记录到问题清单，由开发者决定是否修改代码。

**Q3：审核过程中发现新的代码模块怎么办？**
A3：补充到审核清单中，判断是否需要新增文档或更新现有文档。

**Q4：审核期间代码被更新了怎么办？**
A4：
1. 每日审核前检查`git log -1`，确认代码是否被更新
2. 如果被更新，记录新的commit hash和变更内容
3. 评估变更是否影响已审核的文档
4. 必要时重新审核受影响的文档
5. 在`审核日志.md`中记录代码更新和影响评估

**Q5：被审核的文档在审核期间被修改了怎么办？**
A5：
1. 暂停对该文档的审核
2. 对比修改前后的内容，判断是否影响审核结果
3. 在`审核日志.md`中记录文档修改情况
4. 重新审核该文档
5. 更新审核状态记录表

**Q6：如果Git分支状态为Dirty怎么办？**
A6：建议先提交代码再进行审核。如果必须审核，在审核报告中注明"Dirty状态"，并列出未提交的文件。

**Q7：AI无法确定修复方案怎么办？**
A7：
1. 记录问题并标注"需人工审核"
2. 在`问题清单.md`中详细说明问题的复杂性和不确定性
3. 创建临时建议文件，提供可能的修复方向
4. 不强行修改，等待开发者反馈

**Q8：只审核部分文档怎么办？**
A8：
1. 在`审核状态记录表.md`中，将不审核的文档状态标记为"已跳过"
2. 在`审核报告.md`中注明"本次审核范围为部分文档"
3. 在审核概况中明确列出已审核和未审核的文档清单

**Q9：文档版本号与package.json不一致怎么办？**
A9：
1. 读取`package.json`获取当前版本号
2. 搜索文档中所有版本号引用
3. 如果差异小于1个小版本（如v4.19.0 vs v4.19.1），更新文档中的版本号
4. 如果差异较大（如v4.19.0 vs v5.0.0），记录为重要问题，提醒开发者确认
5. 在`问题清单.md`中详细记录版本号差异情况

**Q10：修复后出现新问题怎么办？**
A10：
1. 在`修复记录.md`中记录新问题的发现
2. 评估新问题的严重程度
3. 如果新问题比原问题更严重，考虑回滚修复
4. 继续修复新问题，直到问题解决
5. 记录整个修复迭代过程
